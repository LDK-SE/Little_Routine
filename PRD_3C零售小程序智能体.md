## 1. 用户角色

| 编号 | 角色 | 身份说明 | 核心诉求 |
|------|------|----------|----------|
| R1 | **老板/店长** | 门店经营决策者 | 看实时毛利、库存水位、员工业绩、资金流水，不被数据错误误导 |
| R2 | **销售员** | 一线接待与开单 | 扫码出库快、查库存准、算提成透明、收款方式灵活 |
| R3 | **仓管员** | 入库验收与盘点 | 扫码入库不重不漏、审核流程清晰、货位一目了然 |
| R4 | **会员（C端消费者）** | 购机与积分用户 | 查积分余额、消费记录、参加老带新活动 |
| R5 | **AI 客服（系统角色）** | 只读查询代理 | 回答库存/积分/订单查询，不操作任何写链路 |

## 2. 核心业务流程图（文字描述）

### 2.1 扫码入库流

```
采购到货 → 仓管员打开小程序"扫码入库" 
→ 摄像头扫描 IMEI 条码 → 系统自动识别机型（匹配 Product_SKU）
→ 仓管员补填：颜色、配置、进货价、渠道、货位
→ 提交 → 状态变为"待审核"
→ 老板/店长在"入库审核"列表逐条审核 
→ 审核通过 → Stock_Ledger 写入（状态=在库）→ 库存 +1
→ 审核驳回 → 退回修改，不入库
```

### 2.2 扫码出库 + 销售单生成流

```
顾客到店购机 → 销售员打开小程序"扫码出库"
→ 扫码 IMEI → 系统校验该串码是否"在库"（乐观锁 version 检查）
  ├─ 不在库/已售 → 弹窗拦截，禁止出库
  └─ 在库 → 自动带出：机型、颜色、配置、移动加权成本
→ 销售员录入：售价、销售员、收款方式（现金/微信/花呗/以旧换新/国补）、顾客手机号
→ 提交 → 事务开始：
  ① Stock_Ledger 状态 → "已销售"
  ② 销售单写入，成本/毛利固化为字段（不可再改）
  ③ 若有关联会员：Point_Ledger 写入积分（实付金额 ×1）
  ④ 若使用积分抵扣：扣减积分余额
→ 事务提交 → 短信异步通知会员（金额 + 积分 + 可用积分）
```

### 2.3 会员积分闭环流

```
【积分获取】
  方式A：消费 → 实付 1 元 = 1 分 → Point_Ledger 写入（绑定订单号 + IMEI）
  方式B：老带新 → 被推荐人注册 + 首单消费 → 双方各得 200 分

【积分使用】
  方式A：抵现 → 收银台勾选"使用积分"→ 100 分抵 1 元 → 实时扣减
  方式B：换购 → 积分 ≥ 3000 可进入换购专区兑换商品

【积分过期】
  按年度滚动 → 每日凌晨 Job 扫描到期积分 → 自动扣减 + 提前 7 天短信提醒
```

### 2.4 AI 智能体查询流

```
用户（老板/会员）→ 小程序 AI 入口 → 文字/语音输入
→ Dify 工作流接收 → Function Calling 解析意图
  ├─ "查库存" → 调用 GET /api/inventory/query（只读 token）
  ├─ "查毛利" → 调用 GET /api/finance/gross-profit（只读 token）
  ├─ "查积分" → 调用 GET /api/member/points（只读 token）
  └─ 意图不明 → 置信度评估
       ├─ ≥ 85% → 返回答案
       └─ < 85% → 转人工客服（生成工单，推送企微通知）
→ API Gateway 层校验：AI token 仅允许 GET，拒绝 POST/PUT/DELETE
```

## 4. 权限体系

| 角色 | 库存查询 | 入库申请 | 入库审核 | 扫码出库 | 售价修改 | 会员查看 | 财务报表 | AI 对话 | 系统设置 |
|------|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|
| 老板/店长 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| 销售员 | ✅ | ❌ | ❌ | ✅ | ✅ | ✅ | ❌ | ✅ | ❌ |
| 仓管员 | ✅ | ✅ | ✅(主管) | ❌ | ❌ | ❌ | ❌ | ✅ | ❌ |
| 会员(C端) | — | — | — | — | — | 仅自己 | — | ✅(仅己数据) | — |
| AI 系统 | 只读 | ❌ | ❌ | ❌ | ❌ | 只读 | 只读 | ✅ | ❌ |

**权限实现原则：**
- 后端 JWT token 中嵌入 `role` + `permissions` 数组
- 每个 API 通过 NestJS Guard 校验角色权限
- AI token 额外携带 `access_level: "readonly"`，在网关层统一拦截非 GET 请求
- 网页端不做权限判断，仅做 UI 显隐控制（安全信任后端）

## 7. 异常场景

### 7.1 库存类异常

| 编号 | 异常场景 | 触发条件 | 系统表现 | 恢复措施 |
|------|----------|----------|----------|----------|
| E-01 | **重复扫码入库** | 同 IMEI 在 Stock_Ledger 已存在 | 后端唯一约束拦截，返回"该串码已入库" + 现有状态 | 仓管核对实物，确认是否串号错误 |
| E-02 | **扫码出库不存在** | IMEI 不在 Stock_Ledger 中 | 返回"该串码未入库，请先扫码入库" | 检查条码是否识别错误 |
| E-03 | **同串码并发出库** | 两个销售员同时扫同一 IMEI 出库 | 乐观锁拦截后提交者，返回"该商品已被售出" | 前一次出库生效，后者弹窗提示 |
| E-04 | **出库时状态非"在库"** | 串码状态为 pending_audit/sold/returned | 拦截，返回"该商品当前状态不允许出库" | 检查串码实际位置 |
| E-05 | **扫码无法识别** | 条码模糊/损坏/不标准 | 提示"无法识别，请手动输入 IMEI" | 手动输入 15 位 IMEI |

### 7.2 交易类异常

| 编号 | 异常场景 | 触发条件 | 系统表现 | 恢复措施 |
|------|----------|----------|----------|----------|
| E-06 | **付款成功但订单失败** | 收款回调到达但数据库事务回滚 | payment_flow 写入但 order 未生成 → 对账脚本发现差异 | 自动退款 or 补单（人工确认） |
| E-07 | **积分扣减事务失败** | 会员积分够但扣减时数据库异常 | 整个销售事务回滚，不扣分也不生成订单 | 重新发起出库 |
| E-08 | **重复收款流水** | 同 payment_no 重复推送 | 唯一键拦截，返回"流水号已存在" | 丢弃重复推送 |
| E-09 | **售价低于成本** | 销售员误录入售价 < 成本快照 | 前端警告（二次确认弹窗），后端记录异常毛利日志 | 老板事后审批或设最低售价限制 |

## 8. MVP 范围

### 8.1 MVP 上线范围（第 1–6 周）

> **一句话原则：一物一码能出入库、能卖、能算利润、能导出库存表。**

#### 必须上线

| 模块 | 功能点 | 验收标准 |
|------|--------|----------|
| **基础** | 手机号 + 验证码登录 | 三端（老板/销售/仓管）可登录，角色区分 |
| **扫码入库** | 扫码 IMEI → 填属性 → 提交审核 → 审核通过 | 仓管可扫、主管可审、串码唯一不重复 |
| **扫码出库** | 扫码校验在库 → 填售价/收款方式 → 生成销售单 | 乐观锁防重复售出、成本自动带出并固化 |
| **库存列表** | 多维筛选（机型/颜色/状态/货位）+ 分页 | 至少支持 4 个筛选维度组合 |
| **库存导出** | 一键导出 Excel | 1000 条库存导出 ≤ 2 秒 |
| **毛利查询** | 今日/本周/本月毛利卡片 + 列表 | 毛利 = 售价 + 国补 - 成本 - 提成（100% 准确） |
| **销售单列表** | 按日期/销售员/收款方式筛选 | 筛选结果与数据库直接查询一致 |

#### MVP 明确不做

| 功能 | 推迟阶段 | 原因 |
|------|----------|------|
| 积分体系建设（1元1分/抵现/换购/推荐） | 第二阶段 | MVP 先跑通交易，积分需要完整会计闭环 |
| 短信通知 | 第二阶段 | MVP 先验证交易核心链路 |
| AI 智能体对话 | 第二阶段 | MVP 期间数据量不足以支撑 RAG，先完成核心数据积累 |
| 多收款方式对账 | 第二阶段 | MVP 先支持录入，不做自动对账 |
| 财务报表（员工业绩/国补汇总） | 第二阶段 | 核心毛利算对即可，报表分类后置 |
| 库存预警/滞销分析 | 第三阶段 | 数据量不足时预警无意义 |
| 销量预测 | 第三阶段 | 至少积累 5 万笔交易数据 |

### MVP 验收检查清单（硬性指标）

- [ ] IMEI 重复入库被唯一约束 + 乐观锁拦截
- [ ] 同 IMEI 并发出库 version 冲突被正确拒绝
- [ ] 销售单生成后，所有请求（含直接 API 调用）均无法修改 cost_price_snapshot 和 gross_profit
- [ ] 库存 Excel 导出列：IMEI、型号、颜色、配置、成本、货位、状态、入库时间
- [ ] 毛利查询结果与手工计算（售价 + 国补 - 成本 - 提成）偏差 = 0
- [ ] 扫码响应 ≤ 300ms，库存列表查询 P95 ≤ 200ms
- [ ] 无 AI 接口接入（MVP 环境不部署 Dify）
